# 产品概述、目标与验收标准

## 1. 方案目标

构建一个支持以下闭环的 AI Native 产品：

```
用户登录
  ↓
上传 PDF
  ↓
异步解析
  ↓
文本切片
  ↓
生成向量
  ↓
生成摘要
  ↓
基于文档问答
  ↓
返回带页码引用的答案
  ↓
记录质量、成本与调用链
```

第一版目标不是构建企业级知识库，而是完成一个：

- 可运行
- 可测试
- 可部署
- 可扩展
- 可验证 AI 输出

的个人 AI 文档工作台。

## 2. 非功能目标

### 2.1 性能目标

| 指标 | MVP 目标 |
| --- | --- |
| 普通 API P95 | < 300ms |
| 创建上传任务 | < 500ms |
| 50 页文本 PDF 处理 | < 60s |
| 问答首 Token 延迟 | < 3s |
| 文件上传成功率 | > 99% |
| 文档处理成功率 | > 95% |
| 引用 ID 有效率 | 100% |

### 2.2 文件限制

第一版限制：

- 文件格式：PDF
- 单文件大小：50MB
- 最大页数：500 页
- 单用户总存储：1GB
- 单文档最大 Chunk：5000

限制必须在以下三个位置**同时**执行：

1. 前端预校验
2. API 创建上传任务时校验
3. Worker 实际解析时再次校验

不能只相信前端。

## 38. 第一版明确不做

为了防止过度设计，MVP 明确不引入：

- 微服务
- Kafka
- Kubernetes
- 独立向量数据库
- 复杂 Agent Runtime
- 多区域部署
- 复杂 ABAC
- 知识图谱
- 多模型智能路由
- 自动工具调用
- 实时协同编辑

这些能力只有在出现明确需求时再演进。

## 39. 最终验收标准

- [ ] 用户可以完成邮箱登录
- [ ] 系统自动创建个人 Workspace
- [ ] 用户可以直传 PDF 到对象存储
- [ ] 完成上传接口具有幂等性
- [ ] 数据库与队列使用 Outbox 保证一致性
- [ ] PDF 处理在独立 Worker 中执行
- [ ] 文档状态和处理阶段可观察
- [ ] Chunk 保存页码和章节信息
- [ ] Embedding 批量生成并保存版本
- [ ] 检索在数据库层完成租户过滤
- [ ] 回答通过 SSE 流式返回
- [ ] 回答包含结构化 Citation
- [ ] Citation 必须通过业务校验
- [ ] 无证据时明确拒答
- [ ] AI 调用记录模型、Token、成本和延迟
- [ ] 文档删除会清除文件、Chunk 和向量
- [ ] 重复 Job 不产生重复数据
- [ ] 单元、集成、E2E 测试通过
- [ ] 本地环境可以一键启动
- [ ] 生产故障具备日志、指标和 Trace

---

这套方案的核心并不是"把 RAG 接起来"，而是建立一个完整的 AI 产品工程闭环：

```
确定性业务系统
+
异步数据处理
+
非确定性 AI 能力
+
权限与安全
+
成本与可观测性
+
Evaluation
```

完成后，DocPilot 不只是一个 PDF 问答 Demo，而是一个具备真实产品基础的 AI Native 系统。
